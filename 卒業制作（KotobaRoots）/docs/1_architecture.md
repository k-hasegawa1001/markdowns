# 🔰 なぜ「いつものコマンド」ではダメなのか？（サーバー構成と役割）

このドキュメントでは、このプロジェクトがなぜ授業で習った構成（`flask run` など）と違うのか、その理由を**「レストランの経営」**に例えて解説します。

専門用語は最後にまとめて解説しているので、まずはイメージを掴んでください。

---

## 1. 「flask run」で公開してはいけない理由

授業では、ターミナルで `flask run` や `python manage.py runserver` と打ってアプリを動かしていました。
これは、料理（プログラム）の味見をするための**「開発用サーバー」**と呼ばれるものです。

これを本番環境（一般公開）で使うのは、**「ワンルームマンションの家庭用コンロでお店を開く」**ようなものです。

### 😱 理由その1：すぐにお店がパンクする（速度・可用性）

家庭用コンロ（開発用サーバー）は、基本的に一口しかありません。
Aさんの料理を作っている間、Bさん、Cさんは**注文すらできずに外で待たされます**（Webサイトがグルグルして繋がらない状態）。

### 😱 理由その2：セキュリティがザル（安全性）

開発用サーバーは、エラーが起きたときに「どこで間違えたか」を親切に表示してくれます。
これは開発者には便利ですが、本番では**「金庫の場所や裏口の鍵の開け方を、お客さんに公開している」**のと同じです。
悪意のある人がこれを見ると、簡単にお店（サーバー）を乗っ取れてしまいます。

---

## 2. 本番環境の「プロの厨房」構成

そこで、本番環境では役割を分担して、本格的なレストランを作ります。

| 役割            | お店の例え               | 実際のソフトウェア名         | 仕事内容                                                                                         |
| :-------------- | :----------------------- | :--------------------------- | :----------------------------------------------------------------------------------------------- |
| **Webサーバー** | **受付・ホールスタッフ** | **Nginx** (エンジンエックス) | お客さん（アクセス）を最初に受け付け、席に案内する。簡単な注文（画像やHTML）は自分で即座に出す。 |
| **APサーバー**  | **厨房・シェフチーム**   | **Gunicorn**, **uWSGI**      | 実際に手の込んだ料理（Pythonなどのプログラム処理）を作ることに専念する。                         |

こうすることで、受付（Nginx）が行列を整理し、厨房（Gunicorn）が並行して料理を作るため、たくさんのお客さんが来てもスムーズに動きます。

---

## 3. なぜ「フロント」と「バック」を分けるのか？

さらにこのプロジェクトでは、**「フロントエンド（画面）」**と**「バックエンド（処理）」**を分けて作っています。
これもレストランで例えると、**「メニュー表」と「料理」の関係**です。

### 🏚️ 授業のやり方（一体型）

授業では、Flaskなどのバックエンドが、HTML（画面）もデータも全部まとめて返していました。
これは、**「シェフが料理を出すたびに、手書きのメニュー表やお皿まで作って渡している」**ようなものです。
シェフの負担が大きく、提供も遅くなります。

### 🏰 今回のやり方（分離型）

今回は以下のように分けました。

1.  **フロントエンド (React/Vueなど):**
    これらは「メニュー表」や「お皿」のような、**内容が変わらないもの（静的ファイル）**です。
    これらは**受付（Webサーバー/Nginx）**の横に置いておき、お客さんが来たら**秒速で渡します。** シェフの手を煩わせる必要はありません。
2.  **バックエンド (Flask/Django API):**
    これらは「ハンバーグ」や「カレー」といった、**注文を受けてから作る料理（動的処理）**です。
    メニューを見て注文が決まったお客さんに対してだけ、**厨房（APサーバー/Flask）**が腕を振るいます。

### ✅ メリット

- **表示が爆速:** メニュー（画面）は受付が即座に出すので、お客さんは待たされません。
- **分業ができる:** 「メニューのデザイン担当（フロント）」と「料理の味付け担当（バック）」で作業を分担できます。

## 4. エンジニア用語の翻訳リスト

エラーが出たときに検索できるように、正しい用語（テクニカルターム）も覚えておきましょう。

| あなたがイメージしたもの | 専門用語 (Technical Term)            | 解説                                                           |
| :----------------------- | :----------------------------------- | :------------------------------------------------------------- |
| **お客さん**             | **リクエスト (Request)**             | スマホやPCからのアクセス。                                     |
| **受付スタッフ**         | **Webサーバー (Reverse Proxy)**      | Nginxなどが担当。セキュリティの盾にもなる。                    |
| **厨房・シェフ**         | **APサーバー (Application Server)**  | Gunicornなどが担当。Pythonなどを動かす。                       |
| **メニュー・お皿**       | **静的ファイル (Static Files)**      | フロントエンドのコード。誰が見ても同じ内容。                   |
| **注文料理**             | **動的コンテンツ (Dynamic Content)** | バックエンドの処理。人によって内容が変わる（マイページなど）。 |

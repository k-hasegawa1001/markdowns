# Gitコンフリクト解決マニュアル

## 1. なぜ（いつ）コンフリクトが起きるのか

Gitは「3ウェイ・マージ」というアルゴリズムで履歴を統合します。基本的には「文脈（前後数行）」が被らなければ自動で解決されますが、以下の場合は人間による判断が必要になります。

### コンフリクトの発生条件

- **同じ行の競合**: 複数のPCで、全く同じ行を異なる内容に書き換えた。
- **文脈の喪失**: 一方のPCで「行を削除（または名前変更）」し、もう一方が「その行のすぐ近く」を編集した。
- **境界線の破壊**: 関数と関数の間の空行（目印）を同時にいじってしまった。

### 豆知識：コンフリクトを防ぐ「設計」の工夫

- **役割ごとのファイル分割**: `auth.py`, `mail.py` のように機能で分ける。
- **緩衝地帯の確保**: 関数間には必ず2行の空行を入れる（PythonのPEP 8など）。
- **共通ルールの徹底**: インデントや空行削除の設定をエディタ（VSCode）で統一する。

## 2. コンフリクト解決の手順

`git sync` （`git pull --rebase`）実行中にエラーが出た際の手順です。

> **⚠️【git syncについて】**
> git syncの中身や導入方法については\
> [【更新者用】git_syncコマンドの作成方法【gitエイリアス】.md](【更新者用】git_syncコマンドの作成方法【gitエイリアス】.md)\
> を参照してください

### STEP 1: 現状の把握

ターミナルのブランチ名が `(main|REBASE 1/1)` のような表示になっていれば、リベース作業が一時停止しています。

### STEP 2: VSCodeマージエディターでの修正

1. 競合しているファイルを開く。
2. 右下の **[マージエディターで開く]** をクリック。
3. 画面を確認：
   - **左上（受信側）**: GitHubから降ってきた内容。
   - **右上（現在）**: 自分の手元でコミットした内容。
   - **下（結果）**: どちらを採用するかボタンで選択。必要なら直接編集。
4. 修正が終わったら **[マージの完了]** を押す。

### STEP 3: 作業の再開

修正したファイルを保存し、ターミナルで以下を順に実行します。

```bash
git add .
git rebase --continue
```

※もし他にも競合コミットがあれば、再度STEP 2に戻ります。全て終われば自動的に通常のブランチに戻ります。

### STEP 4: 仕上げ（プッシュ）

`git sync` は途中で止まっていたので、最後に手動でプッシュして完了です。

```bash
git push origin main
```

---

## 3. 緊急脱出ボタン

「今の修正は一旦白紙に戻したい」「VSCodeの表示がややこしすぎてパニックになった」という時は、迷わず以下を実行してください。

```bash
git rebase --abort
```

**効果**:

- コンフリクトの目印（`<<<<<<<`）が消え、`git sync` を実行する直前の状態に戻ります。
- **自分で書いたコミット済みのコードは消えない**ので安心してください。

# 【備忘録】AWS EC2インスタンスの初期構築・チーム開発環境設定（Ubuntu）

> **📝 読み進める前の注意**
> 本ドキュメント内の `< >` で囲まれた部分（例: `<PCのユーザー名>` や `<任意のサブドメイン>`）は、ご自身のPC環境や取得した値に合わせて書き換えてからコマンドを実行してください。

## 前提条件

- **環境:** AWS Academy (Learner Lab)
- **EC2 OS:** Ubuntu（管理者ユーザー名: `ubuntu`）
- **デバイスA:** 既に独自のキーペアでEC2にSSH接続できている管理者デバイス
- **デバイスB:** 新しく接続を追加したいデバイス

## 1. 複数デバイスから一つのEC2へSSH接続する手順

> **⚠️【重要】チームメンバーを招待する場合の注意点**
> 以下の手順は「管理者自身が別のPCから `ubuntu`（特権アカウント）としてアクセスする」ための手順です。
> チームメンバーを追加する場合は、誤操作によるシステムファイルの破壊を防ぐため、**この手順で `ubuntu` アカウントを共有してはいけません**。必ず「第3章」の手順に従い、個別のユーザーアカウントを作成してください。

### 1-1. デバイスB（新しいPC）でキーペアを生成する

ターミナル（PowerShell等）を開き、以下のコマンドを実行する。

```bash
ssh-keygen -t ed25519
```

※保存場所やパスフレーズは `Enter` を押してデフォルトのまま進める。

### 1-2. デバイスBで「公開鍵」の中身を確認・コピーする

```powershell
# Windowsの場合の確認コマンド
type C:\Users\<PCのユーザー名>\.ssh\id_ed25519.pub
```

表示された `ssh-ed25519 AAA...` から始まる文字列をコピーし、デバイスAへ送る。

### 1-3. デバイスAからEC2に公開鍵を登録する

1. デバイスAからEC2インスタンス（`ubuntu`）にSSH接続する。
2. `~/.ssh/authorized_keys` にデバイスBの公開鍵を追記する。

```bash
nano ~/.ssh/authorized_keys
```

### 1-4. デバイスBからEC2へSSH接続する

```powershell
ssh -i "C:\Users\<PCのユーザー名>\.ssh\id_ed25519" ubuntu@<EC2のIPアドレス>
```

## 2. 無料DDNS（DuckDNS）と systemd を用いたIP自動更新

AWS Academyの仕様上、ラボ起動時にEC2のIPアドレスが変わるため、`systemd` を用いてOS起動時に自動でDuckDNSへIPを通知する仕組みを構築する。

### 2-1. DuckDNSでドメインを取得する

1. [DuckDNS](https://www.duckdns.org/) にログインし、サブドメインを追加する。（例: `<任意のサブドメイン>`）
2. 画面上部の **Token** をメモする。

### 2-2. EC2インスタンスで更新スクリプトを作成する

```bash
mkdir ~/duckdns
cd ~/duckdns
nano duck.sh
```

以下の内容を保存する（DOMAINとTOKENを取得したものに書き換える）。

```bash
#!/bin/bash
DOMAIN="YOUR_DOMAIN"
TOKEN="YOUR_TOKEN"
curl -s "https://www.duckdns.org/update?domains=${DOMAIN}&token=${TOKEN}&ip=" -o /home/ubuntu/duckdns/duck.log
```

実行権限を付与する。

```bash
chmod 700 duck.sh
```

### 2-3. systemdで起動時に自動実行させる

```bash
sudo nano /etc/systemd/system/duckdns-update.service
```

以下の内容を保存する。

```ini
[Unit]
Description=Update DuckDNS IP on boot
After=network-online.target
Wants=network-online.target

[Service]
Type=oneshot
User=ubuntu
ExecStart=/home/ubuntu/duckdns/duck.sh

[Install]
WantedBy=multi-user.target
```

サービスを有効化し、起動テストを行う。

```bash
sudo systemctl daemon-reload
sudo systemctl enable duckdns-update.service
sudo systemctl start duckdns-update.service
cat /home/ubuntu/duckdns/duck.log
```

`OK` と出力されていれば設定完了。

## 3. チーム開発に向けたユーザー管理と共有ディレクトリ（SGID）

ヒューマンエラーによるシステム破壊を防ぐため、日常の開発作業やチームメンバーのアクセスには特権である `ubuntu` ユーザーを使わず、個別のユーザーを作成して共有ディレクトリで作業を行う。

### 3-1. 開発者グループとメンバーアカウントの作成

管理者の `ubuntu` ユーザーでログインし、以下のコマンドを実行する。

```bash
# 1. 開発者共通のグループ（dev）を作成
sudo groupadd dev

# 2. 新しいメンバー（例: member_1）を作成し、devグループに追加
# (-m: ホームディレクトリ作成、-s: ログインシェル指定)
sudo useradd -m -s /bin/bash member_1
sudo usermod -aG dev member_1
```

### 3-2. メンバー用のSSHキー登録

作成したメンバーの領域に、対象者の「公開鍵」を登録する。
（※SSHのセキュリティ仕様により、必ず本人のユーザーに切り替えてから作成すること）

```bash
# メンバーのユーザーに切り替える
sudo su - member_1

# .sshディレクトリを作成し、権限を絞る
mkdir .ssh
chmod 700 .ssh

# authorized_keysを作成し、メンバーから貰った公開鍵を貼り付ける
nano .ssh/authorized_keys

# ファイルの権限を絞り、元のubuntuユーザーに戻る
chmod 600 .ssh/authorized_keys
exit
```

### 3-3. 共有ディレクトリの作成とSGIDの設定

全員が自由にソースコードを編集でき、かつ新規作成したファイルが自動的に `dev` グループの所有になる「魔法の権限（SGID）」を設定する。

```bash
# 1. プロジェクト用の共有ディレクトリを作成
sudo mkdir /project

# 2. 所有者を ubuntu、グループを dev に変更する
sudo chown ubuntu:dev /project

# 3. 権限を「2775（SGID）」に設定する
# (所有者とグループは読書実行可、その他は読実行のみ。新規ファイルはdevグループを引き継ぐ)
sudo chmod 2775 /project
```

## 4. Dockerコンテナと開発環境の連携（インフラとコードの完全分離）

チーム開発におけるセキュリティを確保するため、**「管理者がDockerコンテナを起動・管理し、メンバーは共有ディレクトリでソースコードの編集のみを行う」**運用とする。

### 4-1. メンバーの権限に関するセキュリティ仕様メモ

- **sudo権限なし:** 新規作成したメンバー（`member_1`等）は `sudo` グループに属さないため、デフォルトでシステムを破壊するような特権コマンドは実行できない（奪う設定は不要）。
- **Docker操作権限なし:** メンバーを `docker` グループに追加しないことで、コンテナを利用したルートディレクトリのマウント（特権昇格やバックドアの作成）リスクを完全に排除できる。

### 4-2. docker-compose.yml でのボリュームマウント設定例

管理者が `ubuntu` ユーザーでコンテナを立ち上げる際、メンバーが編集する `/project` ディレクトリをコンテナ内の作業ディレクトリに同期させる。これにより、メンバーがファイルを保存すると即座にコンテナ側のアプリケーションに反映されるため、メンバー自身が `docker` コマンドを打つ必要がなくなる。

```yaml
version: "3.8"
services:
  app:
    # 共有ディレクトリ /project をコンテナ内の /app にマウントする
    volumes:
      - /project:/app
    # その他必要な設定...
```

## 【コラム1】SSH config を使って接続コマンドを極限まで短縮する

毎回長い接続コマンドやドメイン名、鍵のパスを打ち込む手間を省き、**`ssh my-ec2` と打つだけで接続できるようにする**設定。

### 1. `config` ファイルの作成

自身のPCの `~/.ssh/`（Windowsなら `C:\Users\<PCのユーザー名>\.ssh\`）フォルダ内に、拡張子なしで **`config`** というテキストファイルを作成する。

### 2. 設定の記述

以下の内容をご自身の環境に合わせて書き換え、保存する。

```text
Host my-ec2
    HostName <任意のサブドメイン>.duckdns.org
    User ubuntu
    IdentityFile ~/.ssh/id_ed25519

# チームメンバーとして接続する場合の設定例
Host dev-env
    HostName <任意のサブドメイン>.duckdns.org
    User member_1
    IdentityFile ~/.ssh/id_ed25519
```

### 3. 短縮コマンドで接続する

以降は、ターミナルで以下のコマンドを打つだけでEC2に接続できる。

```powershell
ssh my-ec2
```

## 【コラム2】SSH接続とコンテナ起動の「リアルタイム性」について

クラウド環境で複数人が同時に開発を行う際、「自分が接続したタイミング」と「管理者がコンテナを起動したタイミング」の前後関係を気にする必要はありません。

SSH接続は「手元のPCに環境を同期（ダウンロード）する」ものではなく、**「全員で遠隔地にある1台のパソコン（EC2インスタンス）の画面に同時にログインして、同じOSを操作している状態」**です。

そのため、以下のようなケースでも完全にリアルタイムで状況が共有されます。

- **行動が早いメンバーが先に接続して待機している場合:** その後から管理者がログインして `docker-compose up` を実行した瞬間、すでに接続済みのメンバーの環境からも即座にコンテナ（Webアプリ等）が利用可能になります。入り直す必要は一切ありません。
- **ファイルの同時編集:** メンバーが共有ディレクトリ内のソースコードを保存した瞬間、稼働中のコンテナに即座に変更が反映されます。

全員が「物理的に同じ1つの部屋（Ubuntu OS）」にいるというイメージを持つと、システム管理やトラブルシューティングが非常に分かりやすくなります。

## 【コラム3】究極の証拠保全：CloudWatch Logsによる外部退避

`root` 権限を奪取した悪意あるユーザーが `sudo rm -rf /`等でログファイルごと証拠隠滅を図る「完全犯罪」を防ぐためのプロのベストプラクティスです。

**仕組み：**
EC2インスタンス内に「CloudWatch Agent」を導入し、`/var/log/auth.log`（セキュリティログ）等を監視させます。コマンドが実行されOSが破壊されるより早い**ミリ秒単位の速度でAWSの安全な別サーバーへログを転送（退避）する**ことで、EC2が沈黙した後でも「誰が引き金を引いたか」を確実に特定できます。（AWS Academyの無料枠5GB内で実質無料で運用可能です）。

## 【コラム4】次世代の接続方式「AWS Systems Manager (Session Manager)」

近年、企業で最も推奨されている「SSH（ポート22番）を使わない」接続方式です。チーム全員がAWSコンソールにログインできる環境であれば、この方式が最も安全です。

**メリット：**

1. **ポートの完全閉鎖:** セキュリティグループでポート22を閉じることができるため、外部からのログイン攻撃を100%遮断できます。
2. **鍵管理の廃止:** 公開鍵（`.pub`）の作成や受け渡しが不要になり、AWSコンソールの権限だけでブラウザから安全にターミナルへ接続できます。
3. **完全な操作録画:** （オプション設定）ターミナルで打ち込んだ全ての操作と出力結果をS3やCloudWatch Logsに「録画」として保存でき、強力な監査ログとなります。

**⚠️注意点：**
この方法はIAMロールに関する操作（IAMユーザーの作成・招待など）が必要ですが、AWS AcademyのLearner Lab環境では、学生に与えられている権限が厳しく制限されているため、システムレベルでIAMロール操作権限が剥奪されています。\
しかしこの方法は企業で最近ベストプラクティスとして台頭してきているものであるため、就職後などにこちらの環境での作業を求められるかもしれません。\
また、権限分離の観点でも非常に大きな落とし穴があるため、もし個人アカウントなどで実験を行う場合には「**Run As**」 という設定を行うなど、十分にリスクやその対処法について調べてから行ってください。

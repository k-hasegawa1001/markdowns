# 🐳 Dockerを用いたモダンな開発ワークフローとDocker Composeの活用

## 1. はじめに：なぜ今、Dockerで開発するのか？

「自分のPCでは動いたのに、あの人のPCでは動かない」「本番環境にデプロイしたらエラーが出た」
開発現場でよくあるこれらの悩みは、**Docker**を正しく導入することで過去のものになります。

Dockerは単なる「本番環境へのデプロイツール」ではありません。**「開発体験（Developer Experience）」を劇的に向上させるための最強のツール**です。

本ドキュメントでは、開発効率を最大化するためのDocker活用法と、複数コンテナを束ねるDocker Composeの必要性について解説します。

---

## 2. Dockerを用いた開発の基本スタイル

従来の開発では、ローカルPCに言語ランタイム（Node.js, Python, Go, Javaなど）やデータベースを直接インストールしていました。しかし、Dockerを用いた開発ではアプローチが異なります。

### 🔑 核心技術：バインドマウント（Bind Mount）

開発用コンテナにおける最大のポイントは、**ソースコードをコンテナの中にコピーしない**ことです。
代わりに、ローカルのソースコードディレクトリをコンテナ内のディレクトリと同期させる**バインドマウント**という機能を使います。

- **ローカルPC:** お気に入りのエディタ（VSCodeなど）でコードを編集・保存。
- **コンテナ:** 変更を即座に検知し、実行・テストを行う。

これにより、**「編集はローカル、実行はコンテナ」**といういいとこ取りが可能になります。

---

## 3. 言語タイプ別のDocker開発メリット

### 🅰️ インタプリタ言語の場合

（例：JavaScript/Node.js, Python, PHP, Rubyなど）

コードをコンパイルせずに実行する言語では、Dockerのメリットは「環境の汚染防止」と「バージョン管理」にあります。

- **メリット:**
  - **ローカル環境を汚さない:** `node_modules` や `pip` パッケージでPCが散らかりません。プロジェクトごとに完全に独立した環境を持てます。
  - **チーム全員が同じバージョン:** `python:3.9` や `node:18` など、Dockerイメージでバージョンを固定するため、「微妙なバージョン違いによるバグ」がゼロになります。
- **開発フロー:**
  - ローカルでコードを保存 → コンテナ内のホットリロードツール（`nodemon`など）が検知 → 即座に反映。

### 🅱️ コンパイル言語の場合

（例：Go, Java, Rust, C#など）

「コンパイル言語はビルドが必要だからDockerだと遅いのでは？」というのは過去の誤解です。現在は「**コンテナ内ホットリロード**」が主流です。

- **開発フローの革命:**
  1.  開発用コンテナには、コンパイラと**ホットリロードツール**（GoのAir, JavaのDevToolsなど）を入れておく。
  2.  ローカルでコードを保存する。
  3.  コンテナ内のツールが変更を検知し、**コンテナ内で高速に自動再ビルド・再起動**を行う。
- **メリット:**
  - **ローカルにコンパイラ不要:** 開発者は、手元のPCに言語（GoやRust本体）をインストールする必要すらありません。Dockerさえあれば、今日入った新人もすぐに開発に参加できます。
  - **本番環境とのOS一致:** WindowsやMacで開発していても、コンテナ内はLinuxです。パス区切り文字やシステムコールの違いによる「本番だけで起きるバグ」を未然に防げます。

## 4. なぜ「Docker Compose」が必要なのか？

単体の `docker run` コマンドでもコンテナは起動できますが、実際のシステム開発では限界があります。そこで登場するのが **Docker Compose** です。

### 🚀 複数のコンテナを「オーケストレーション（指揮）」する

Webシステムは通常、単独では動きません。「Webサーバー(Nginx)」「アプリケーション(AP)」「データベース(DB)」「キャッシュ(Redis)」などが連携して動作します。

Docker Composeを使うと、これら全ての構成を `docker-compose.yml` という1つのファイルに定義できます。

#### Docker Composeがない世界 😱

```bash
# ネットワークを作って、DBを起動して、パスワードを設定して、アプリを起動して、リンクして...
$docker network create my-net$ docker run -d --name db --network my-net -e POSTGRES_PASSWORD=secret postgres
$ docker run -d --name app --network my-net -p 8080:8080 my-app
# (毎回このコマンドを覚えますか？)
```

#### Docker Composeがある世界 😎

```bash
# たったこれだけ。
$ docker compose up
```

### 🔗 サービスディスカバリ（自動名前解決）

Docker Composeの最大の利点は、**「IPアドレスを管理しなくていい」**ことです。

`docker-compose.yml` にサービス名（例: `db`, `web`, `redis`）を定義すると、コンテナ内部ではその名前がドメイン名として使えます。

- アプリの設定ファイルには `DB_HOST=192.168.x.x` ではなく、 `DB_HOST=db` と書くだけで繋がります。

## 5. まとめ：目指すべき開発スタイル

私たちが目指すのは、以下のような開発体験です。

1.  **Clone & Up:** リポジトリをクローンし、`docker compose up` を叩く。
2.  **Code:** 好きなエディタで開発する。保存すれば勝手に反映される。
3.  **No Environmental Issues:** 「環境構築で1日終わる」「環境差異で動かない」という時間をゼロにする。

DockerとDocker Composeは、インフラ担当者のためのツールではなく、**アプリケーション開発者が「コードを書くこと」に集中するためのツール**です。

ぜひ、この快適な環境を使いこなしていきましょう！

# プロジェクト セキュリティ＆環境変数 管理ガイドライン

本ドキュメントは、当プロジェクトにおける機密情報（APIキー、パスワード等）の安全な取り扱いと、ローカル開発環境から本番環境へのシームレスな移行を実現するための標準的な手順を定めたものです。開発メンバー全員がこのルールを理解し、遵守してください。

---

## 1. セキュリティの重要性と情報漏洩の危険性

機密情報の漏洩は、プロジェクトの存続を脅かす致命的なインシデントに直結します。特に、OpenAIやAWSなどの従量課金制クラウドサービスの認証情報が漏洩した場合、数時間で数百万円規模の不正利用被害が発生するリスクがあります。

### ❌ GitHub（パブリック領域）への共有が危険な理由

- **即時検知と悪用:** GitHubの公開リポジトリは、悪意あるBotによって24時間365日監視されています。Pushした瞬間にキーは自動検知され、不正利用が始まります。
- **履歴の永続性:** Gitは一度コミットしてしまったファイルを永遠に記憶します。後から `git rm` で削除したり `.gitignore` に追加したりしても、過去のコミット履歴から誰でも閲覧可能です。

### ❌ チャットアプリ（Slack, Discord, LINE等）での共有が危険な理由

- **アクセス制御の喪失:** チャットツールに平文で機密情報を送信すると、誰が閲覧権限を持っているかの追跡・制御が不可能になります。
- **アカウントの乗っ取りリスク:** メンバーの端末紛失やアカウント乗っ取りが発生した場合、過去のログからプロジェクト全体の機密情報が芋づる式に流出します。

## 2. GitHubを安全に使うための `.gitignore` 運用

Gitの管理下から特定のファイルを除外するために、プロジェクトのルートディレクトリに必ず `.gitignore` ファイルを配置します。

### 記述すべき3つの主要カテゴリ

1. **機密情報・クレデンシャル（最重要）:** `.env` などの環境変数ファイル、`.pem` などの鍵ファイル。
2. **OS・エディタの自動生成ファイル:** `.DS_Store` (macOS)、`Thumbs.db` (Windows)、`.vscode/` (エディタ設定) など。
3. **言語・フレームワーク固有のファイル:** `node_modules/`、`__pycache__/`、`venv/` など。

### 💡 推奨ツール：gitignore.io の活用

プロジェクトの立ち上げ時は自動生成ツールの利用を推奨します。

1. [gitignore.io](https://www.toptal.com/developers/gitignore) にアクセス。
2. 使用する技術スタック（例：`Python`, `Flask`, `Docker` 等）を入力して生成。
3. 生成されたテキストを `.gitignore` にコピー＆ペースト。
4. **最後に手動で `.env` 関連の除外設定が含まれているか目視確認する。**

## 3. `docker-compose.yml` のベストプラクティス

`docker-compose.yml` はインフラの構成図としてGitにコミットするため、**絶対にパスワード等を直接（ハードコード）書き込んではいけません。**
必ず「変数展開」を使用し、値は外部（`.env`等）から読み込む設計にします。

**✅ 正しい記述例：**

```yaml
services:
  db:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: ${DB_ROOT_PASSWORD} # 直接書かず変数で受け取る
```

## 4. チーム内での安全な `.env` 共有方法（`.env.example` の運用）

「アプリを動かすためには何の変数が必要か」という仕様を共有するために、**中身がダミーのひな形ファイル `.env.example` を作成し、Gitにコミットします。**

### `.env.example` の作成例

```text
# ローカル開発用のDB設定（デフォルトのまま使用してください）
DB_ROOT_PASSWORD=dummy_password_for_local

# 各自で取得したテスト用のAPIキーを入力してください
OPENAI_API_KEY=
```

### 新規参画メンバーの環境構築フロー

1. リポジトリを `git clone` する。
2. `cp .env.example .env` を実行し、自分専用の `.env` を作成する。
3. `.env` を開き、空欄になっているAPIキー等に、各自で発行したテスト用キーを入力する。
4. `docker compose up -d` を実行する。

## 5. 開発環境と本番環境の切り替え（学習・制約環境向け）

ローカル環境と本番サーバーで設定値を変更するため、複数の環境変数ファイルを使い分けます。

- **`.env.development`**：ローカル開発用（ダミーDBパスワード、テスト用APIキー等）
- **`.env.production`**：本番環境用（本番用APIキー等）

`docker-compose.yml` の記述は変更せず、起動コマンドで読み込むファイルを切り替えます。

```bash
# ローカル開発環境
docker compose --env-file .env.development up -d

# 本番環境（※本番用ファイルをSCP等でサーバーに配置した場合）
docker compose --env-file .env.production up -d
```

※注意：本番サーバーに物理的な `.env` ファイルを配置する運用は、AWS Academyなどの権限（IAM）制約がある環境下での特例措置（妥協案）です。

## 6. 実務における本番環境のベストプラクティス（AWS Secrets Manager）

商用サービス等の本格的な実務環境では、セキュリティと自動化の観点から**「本番サーバー（EC2等）のディスク上に `.env` ファイルを物理的に作成・配置すること」自体を禁止（アンチパターン）**とします。

代わりに、クラウドのセキュアな機密情報管理サービスである **AWS Secrets Manager** を活用し、「ファイルには残さず、メモリ上で直接読み込む」完全自動化の構成をとります。

### 実務での正しい運用フロー

1. **機密情報の隔離:** インフラ管理者が、本番用のAPIキーやDBパスワードをAWS Secrets Manager（金庫）に直接登録します。
2. **IAMロールによる権限付与:** EC2インスタンスに対し「特定の金庫だけを開けてよい」という最小権限のIAMロールを付与します。
3. **サーバー起動時の自動取得:** EC2が起動する際（またはアプリが立ち上がる際）、AWS CLIやプログラム内のSDK（boto3等）を経由して Secrets Manager から動的に値を取得します。

**✅ AWS CLIによる自動取得のイメージ（EC2の起動スクリプト等で実行）**

```bash
# Secrets Managerから本番用キーを取得し、直接環境変数としてエクスポートする
export OPENAI_API_KEY=$(aws secretsmanager get-secret-value --secret-id MyApp/Production/OpenAIKey --query SecretString --output text)

# .envファイルを介さずにDockerコンテナを起動
docker compose up -d
```

このアーキテクチャにより、本番サーバーにSSHログインして手作業を行う必要がなくなり、オートスケーリング（サーバーの自動増減）への対応や、内部犯行リスクの極小化を実現できます。

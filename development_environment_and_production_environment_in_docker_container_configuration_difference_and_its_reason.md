# 開発環境と本番環境におけるDockerコンテナ構成の違いとその理由

Webアプリケーション開発において、ローカル開発環境（手元のPC）と本番・結合テスト環境（EC2などのリモートサーバー）では、求められる要件が根本的に異なります。そのため、Dockerを用いたコンテナ構成もそれぞれの目的に合わせて変化させる必要があります。

本資料では、フロントエンドにReact、バックエンドにAPIサーバー（Python等）、データベース、そして厳しいポート制限（80, 443, 22のみ開放）がある本番環境を想定し、両環境のアーキテクチャの違いとその理由を解説します。

---

## 1. ローカル開発環境の構成

**■ 目的：開発スピードと利便性の最大化**
ローカル環境の最大の目的は「コードの変更を即座に画面に反映させ（ホットリロード）、効率よく開発を進めること」です。

### アーキテクチャ図解（概念）

- **Nginx (ポート8080公開):** フロントエンドとバックエンドへの通信を振り分ける（本番環境の挙動を模倣）。
- **Frontend (Node.jsコンテナ):** ソースコードをバインドマウントし、Reactの開発サーバー（`npm start`等）を稼働させる。
- **Backend (APIコンテナ):** ソースコードをバインドマウントし、APIサーバーを稼働させる。
- **Database (DBコンテナ):** バックエンドからのデータ保存・取得を担う。

### 設定のポイントと理由

1. **バインドマウント（Volumes）の活用**
   ホストOS（手元のPC）のソースコードディレクトリをコンテナ内部にマウント（同期）します。これにより、エディタでコードを上書き保存した瞬間にコンテナ内のプロセスが変更を検知し、即座に画面が更新されます。
2. **フロントエンドは「開発用サーバー」を起動**
   ローカルでは静的ファイルへのビルド（`npm run build`）は行いません。都度ビルドを行うと確認に数分かかり、開発効率が著しく低下するためです。Node.jsイメージを使用し、開発サーバーを起動し続けます。

---

## 2. 本番・結合テスト環境（EC2等）の構成

**■ 目的：セキュリティ、パフォーマンス、リソース効率の最大化**
本番環境では、ファイルの変更検知は不要です。代わりに、外部からの攻撃を防ぐ堅牢なネットワーク設計と、限られたサーバースペックを最大限に活かす軽量化が求められます。

### アーキテクチャ図解（概念）

- **Nginx (ポート80/443のみ公開):** 門番（リバースプロキシ）兼、静的ファイルの配信サーバー。
- **Backend (APIコンテナ):** APIリクエストを処理。外部ネットワークからは隔離。
- **Database (DBコンテナ):** バックエンドからのみアクセス可能な閉鎖網に完全隔離。
- **Dozzle (ログ可視化コンテナ):** 開発チームがブラウザからログを確認するためのツール。

### 設定のポイントと理由

1. **Nginxによる「マルチステージビルド」でのフロントエンド配信**
   本番環境では、Reactのコードはコンテナビルド時に静的ファイル（HTML, CSS, JS）へ変換（ビルド）されます。そして、生成されたファイルのみをNginxコンテナに配置して配信します。Node.jsの実行環境が不要になるため、コンテナサイズが劇的に軽く、かつセキュアになり、Nginx本来の超高速な静的ファイル配信の恩恵を受けられます。
2. **厳しいポート制限（80, 443）のクリアとリバースプロキシ**
   学校や企業のネットワークでは、セキュリティのため特定のポートしか通信が許可されていません。そのため、EC2の入り口であるNginx（80番ポート）だけを外部に公開します。
   NginxはアクセスされたURL（例: `/api` や `/logs`）に応じて、Docker内部ネットワークだけで繋がっているBackendやDozzleへ通信を安全に中継（リバースプロキシ）します。
3. **データベースのネットワーク完全分離**
   Docker Composeの機能を使って「Nginxと通信するネットワーク（web-network）」と「DBと通信するネットワーク（db-network）」を分割します。バックエンドコンテナのみを両方のネットワークに所属させることで、外部からDBへの不正アクセスの経路を物理的に遮断します。
4. **IAM権限・SSH不要のログ監視（Dozzle）**
   本番環境でエラーが発生した際、全員がサーバーにSSH接続してログを確認するのはセキュリティリスクであり、運用上も現実的ではありません。Dozzleコンテナを配置し、Nginx経由（例: `http://<IP>/logs/`）でブラウザからリアルタイムログを見られるようにすることで、インフラ管理者の負担をゼロにします。

---

## 3. 両環境の比較まとめ

| 項目               | ローカル開発環境                   | 本番・結合テスト環境 (EC2)                  |
| :----------------- | :--------------------------------- | :------------------------------------------ |
| **フロントエンド** | Nodeコンテナで開発サーバーを起動   | Nginxコンテナ内で静的ファイルとして高速配信 |
| **コードの反映**   | バインドマウント（保存で即時反映） | CI/CDによるコンテナの再ビルドとデプロイ     |
| **ポートの公開**   | 開発用の任意のポート（8080など）   | Nginxの80番（および443番）のみ              |
| **ログの確認**     | 手元のターミナルで確認             | Dozzleを通じ、ブラウザ（GUI）で確認         |

このように、フェーズによってDockerの使い方を切り替えることで、「開発のしやすさ」と「本番の安全性・高速性」を両立させることができます。

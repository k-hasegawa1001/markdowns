## 事象1：OS起動時にコンテナが勝手に起動してしまう

### 1-1. きっかけ

EC2インスタンス（Ubuntu）を起動した際、自身で `docker-compose up -d` コマンドを実行していないにもかかわらず、バックグラウンドでWordPressやMySQLのコンテナが自動的に起動状態になっていた。

### 1-2. 原因：Dockerデーモンと再起動ポリシー（Restart Policy）

この現象は、OSの自動実行スクリプト（systemdなど）が誤作動したわけではなく、**Dockerの標準機能である「再起動ポリシー」**による正しい挙動である。

**【Dockerデーモンの内部動作】**

1. EC2インスタンスが起動すると、OSのプロセスとして **Dockerデーモン（`dockerd`）** が自動的に立ち上がる。
2. Dockerデーモンは起動直後、ローカルに保存されているすべてのコンテナの状態（メタデータ）をスキャンする。
3. その際、`docker-compose.yml` において **`restart: always`** または **`restart: unless-stopped`** と定義されて作成されたコンテナを発見すると、Dockerデーモンは「このコンテナは常に稼働しているべきだ」と判断し、ユーザーの指示を待たずに自律的にコンテナのプロセスを再起動させる。

### 1-3. 対処法：ポリシーの無効化とコンテナの再作成

開発環境などで「自分が必要な時だけ手動で起動・停止したい（リソースを節約したい）」場合は、この設定を無効化する。

**1. `docker-compose.yml` の修正**
対象となるすべてのサービス（コンテナ）の記述から `restart: always` を削除するか、明示的に `"no"` を指定する。

```yaml
services:
  wordpress:
    image: wordpress:latest
    restart: "no" # ダブルクォーテーションで囲む
    # ...
  db:
    image: mysql:5.7
    restart: "no"
    # ...
```

**2. 変更の適用（コンテナの再作成）**
ymlファイルを修正しただけでは現在稼働中のコンテナには反映されないため、一度コンテナを破棄して作り直す。

```bash
docker-compose down
docker-compose up -d
```

これで、次回以降のEC2インスタンス起動時はコンテナが停止状態のまま待機するようになる。

## 事象2：docker-compose down がネットワークエラーで失敗する

### 2-1. きっかけ

事象1の対処として `docker-compose.yml` を修正し、`docker-compose down` でコンテナを停止・削除しようとしたところ、「対象のネットワークが見つからない」といったエラーが発生し、コマンドが弾かれてしまった。

### 2-2. 原因：ディレクトリ名（プロジェクト名）の変更による紐付けの喪失

このエラーは、Docker Composeがリソース（コンテナ、ネットワーク、ボリューム）を管理・識別するための**「プロジェクト名」の仕様**に起因する。

**【Docker Composeの内部仕様】**

- Docker Composeは、リソースを作成する際、デフォルトで **`docker-compose.yml` が置かれている親ディレクトリの名前** を「プロジェクト名」として採用する。
- 例えば、`/home/ubuntu/sample_docker_compose/` 配下で `up` を実行した場合、作成されるネットワーク名やコンテナ名の先頭には `sample_docker_compose_...` というプレフィックス（接頭辞）が付与され、これがメタデータとして保存される。
- もし、稼働中に**ディレクトリ名を変更**したり、**ymlファイルを別のディレクトリに移動**した上で `down` を実行すると、Docker Composeは「現在の（新しい）ディレクトリ名」をベースにネットワークを探しに行くため、「前回作成されたネットワークが存在しない（名前が違う）」と判断してエラー終了してしまう。

### 2-3. 対処法：手動でのコンテナ強制削除とネットワークのクリーンアップ

`docker-compose` コマンドによる一括管理の紐付けが切れてしまった場合は、Dockerの基本コマンド（`docker`）を使用して、個別のリソースを手動で削除する。

> **⚠️ 注意（データ消失リスク）**
> コンテナを削除すると、内部のデータ（DBのレコード等）も消滅する。データを永続化したい場合は、事前に `docker-compose.yml` でボリュームマウント（ホスト側ディレクトリとの同期）を設定しておく必要がある。

**手順1：稼働中のコンテナ名（またはID）を特定する**

```bash
docker ps
```

※右端の `NAMES` 列に表示されている、旧ディレクトリ名が含まれたコンテナ名をメモする。

**手順2：コンテナを個別に停止する**

```bash
docker stop <WordPressのコンテナ名> <MySQLのコンテナ名>
```

**手順3：コンテナを個別に削除する**

```bash
docker rm <WordPressのコンテナ名> <MySQLのコンテナ名>
```

**手順4：使われなくなった古いネットワーク（残骸）を一掃する**
コンテナを削除しても、旧ディレクトリ名で作成されたネットワークが残っているため、以下のコマンドで不要なネットワークをまとめて削除する。（現在稼働中の他のコンテナのネットワークには影響しない）

```bash
docker network prune
# 警告文が出たら 'y' を入力してEnter
```

**手順5：正しいディレクトリ名で再構築する**
正しいディレクトリに配置した修正済みの `docker-compose.yml` を用いて、改めてコンテナを起動する。

```bash
cd /home/ubuntu/<正しいディレクトリ名>
docker-compose up -d
```

これにより、新しいディレクトリ名をプロジェクト名とした正しいネットワークとコンテナが再構築される。

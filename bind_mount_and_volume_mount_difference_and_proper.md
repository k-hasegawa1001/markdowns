# Dockerのデータ管理：バインドマウントとボリュームマウントの違いと使い分け

Dockerコンテナは一時的な実行環境であるため、コンテナを破棄すると内部のデータも失われます。データを永続化（保存）し、コンテナの再作成後も利用できるようにするために、主に**バインドマウント**と**ボリュームマウント**の2つの手法が用いられます。

---

## 1. バインドマウント (Bind Mounts)

ホストマシンのファイルシステム上の特定のディレクトリやファイルを、コンテナ内に直接マウントする方式です。

### 特徴

- ホスト側の絶対パスまたは相対パスを直接指定してマウントします。
- **ディレクトリ単位だけでなく、ファイル単位（特定の1ファイルのみ）でのマウントも可能です。**
- ホスト側のファイルシステム（ディレクトリ構造）に強く依存します。
- コンテナ外のプロセスやツール（IDEやエディタなど）から、マウントしたファイルを直接編集・確認できます。

### 使いどころ

- **ローカル開発環境でのソースコード共有:** ホスト側でコードを編集し、コンテナ側のアプリケーションに即座に反映（ホットリロード）させたい場合。
- **特定の設定ファイルのみの注入:** コンテナ内のディレクトリ全体を上書きせず、ホスト側で管理している設定ファイル（例: `nginx.conf` など）の1ファイルだけをピンポイントで適用したい場合。

### 注意点

- **ポータビリティの低下:** ホストマシンのOS（Windows, macOS, Linux）やディレクトリ構成に依存するため、他の環境にそのまま持ち込む（共有する）ことが難しい場合があります。
- **権限（パーミッション）問題:** ホストとコンテナ間で実行ユーザーの権限設定が異なる場合、ファイルの読み書きエラーが発生することがあります。
- **ホスト環境への影響:** コンテナ内からホスト側の重要なファイルを意図せず変更・削除できてしまうリスクがあります（読み取り専用の `ro` オプションをつけることで回避可能）。

### 記述例

```bash
# docker runコマンドの場合
docker run -v <ホストのディレクトリまたはファイルパス>:<コンテナ内のパス> <イメージ名>

# docker-compose.ymlの場合
services:
  <サービス名>:
    image: <イメージ名>
    volumes:
      - <ホストのディレクトリまたはファイルパス>:<コンテナ内のパス>
```

---

## 2. ボリュームマウント (Volumes)

Dockerエンジンが管理するホスト上の専用領域をコンテナにマウントする方式です。Docker公式が最も推奨しているデータ永続化の方法です。

### 特徴

- Dockerがデータの保存場所を完全に管理するため、ホストのファイルシステム構造から分離されます。
- **マウントの単位は「ディレクトリ（領域）単位」のみであり、特定のファイル単体だけをマウントすることはできません。**
- 複数のコンテナ間で安全かつ簡単にデータを共有できます。
- バックアップ、復元、マイグレーション（移行）が容易です。

### 使いどころ

- **データベースのデータ永続化:** `<データベース名>` などのデータを安全に保存したい場合（本番環境・開発環境問わず推奨）。
- **コンテナ間のデータ共有:** 複数のコンテナから同じデータ領域にアクセスする必要がある場合。
- **本番環境での運用:** ホストのOS環境に依存せず、安全かつ安定してデータを管理したい場合。

### 注意点

- **ホスト側からの直接編集が困難:** Docker Desktop（macOS/Windows）などを使用している場合、ボリュームの実体は隠蔽された仮想マシン内にあるため、ホスト側からファイルを直接開いて編集することが困難です。
- **不要なボリュームの蓄積:** コンテナを削除してもボリュームは自動的には削除されません。手動で削除（`docker volume rm <ボリューム名>` や `docker volume prune`）を行わないと、ディスク容量を圧迫し続けます。

### 記述例

```bash
# docker runコマンドの場合
docker run -v <ボリューム名>:<コンテナ内のディレクトリパス> <イメージ名>

# docker-compose.ymlの場合
services:
  <サービス名>:
    image: <イメージ名>
    volumes:
      - <ボリューム名>:<コンテナ内のディレクトリパス>

volumes:
  <ボリューム名>: # ボリュームの定義（必須）
```

## 3. 比較まとめ

| 項目                     | バインドマウント                             | ボリューム（推奨）                 |
| :----------------------- | :------------------------------------------- | :--------------------------------- |
| **マウントの単位**       | ディレクトリ、および**ファイル単位**         | **ディレクトリ単位**のみ           |
| **保存場所**             | ホストマシンの任意の場所                     | Dockerが管理する専用領域           |
| **ホストへの依存性**     | 高い（OSやパス構成に依存）                   | 低い（Dockerが吸収）               |
| **ホストからのアクセス** | 非常に容易                                   | 困難（Linux以外は仮想マシン内）    |
| **主な用途**             | 開発時のソースコード共有、設定ファイルの配置 | DBデータの永続化、本番環境での運用 |
| **ポータビリティ**       | 低い                                         | 高い                               |

## 4. 補足：よくあるトラブルと運用手順

### 4.1. バインドマウント時のパーミッション（権限）エラーと回避方法

バインドマウントを使用する際、特にLinux環境で頻発するのが「コンテナ内からファイルに書き込めない」「ホスト側からファイルを開けない」といったパーミッションエラーです。

**原因:**
ホスト側のディレクトリの所有者（UID/GID）と、コンテナ内でアプリケーションを実行しているユーザー（rootやnode, nginxなど）が異なるために発生します。

**回避方法:**

1.  **コンテナの実行ユーザーをホスト側に合わせる（推奨）**
    コンテナを起動する際、ホスト側のユーザーIDを指定して実行することで権限の不一致を防ぎます。

    ```bash
    # docker-compose.yml の記述例
    services:
      <サービス名>:
        image: <イメージ名>
        user: "<ホストのUID>:<ホストのGID>" # 例: "1000:1000"
        volumes:
          - <ホストのディレクトリパス>:<コンテナ内のディレクトリパス>
    ```

2.  **ボリュームマウントへの切り替え**
    ソースコードの直接編集が不要なデータ（データベースのデータなど）であれば、そもそもバインドマウントを使用せず、Dockerが権限を適切に管理してくれる「ボリュームマウント」に切り替えるのが最も安全で確実です。

### 4.2. ボリュームのバックアップと復元手順

ボリュームの実体はDockerが管理する仮想環境内に隠蔽されているため、ホストから直接フォルダをコピーしてバックアップすることは推奨されません。
一時的なコンテナ（`alpine`や`ubuntu`など）を立ち上げ、そこを経由してアーカイブ（`tar`ファイルなど）として入出力するのが標準的な手法です。

#### バックアップ手順

対象のボリュームと、バックアップファイルを保存するホストのディレクトリを両方マウントした一時コンテナを起動し、ボリュームの中身を圧縮して保存します。

```bash
docker run --rm \
  -v <対象ボリューム名>:/volume_data \
  -v <ホストのバックアップ保存先パス>:/backup \
  alpine \
  tar cvf /backup/<バックアップファイル名>.tar -C /volume_data .
```

_※ `--rm` オプションにより、バックアップ処理（tarコマンド）が完了するとこの一時コンテナは自動的に削除されます。_

#### 復元（リストア）手順

新しいボリュームを作成し、バックアップ時と同様に一時コンテナを起動して、アーカイブを展開（解凍）します。

```bash
# 1. 復元先の新しいボリュームを作成（既に存在する場合は省略可）
docker volume create <復元先ボリューム名>

# 2. バックアップファイルを展開して復元
docker run --rm \
  -v <復元先ボリューム名>:/volume_data \
  -v <バックアップファイルがあるホストのパス>:/backup \
  alpine \
  tar xvf /backup/<バックアップファイル名>.tar -C /volume_data
```

このように、一時コンテナを「データの運び屋」として利用することで、ホストOSに依存せず安全にデータの移行やバックアップが可能になります。

### 4.3. ホストOSのユーザーID（UID）とグループID（GID）の確認方法

前項のバインドマウント時のパーミッションエラーを防ぐため、`docker-compose.yml` 等に指定するホスト側のUIDとGIDを確認する手順です。
※この権限設定は、主にLinux、macOS、およびWindowsのWSL2環境で重要になります。

#### Linux / macOS / WSL2環境 の場合

ターミナル（端末）を開き、以下のコマンドを実行します。

**現在のユーザーのUIDとGIDをまとめて確認する**

```bash
id
```

_出力例: `uid=1000(<ユーザー名>) gid=1000(<グループ名>) groups=...`_
この出力結果から、UIDは `1000`、GIDは `1000` であることが分かります。

**UIDのみを抽出して確認する**

```bash
id -u
```

**GIDのみを抽出して確認する**

```bash
id -g
```

取得した数値を、`docker-compose.yml` の `user: "<UID>:<GID>"` の部分に当てはめて使用してください。

#### Windows環境 の場合

Windows標準のファイルシステムはLinuxのUID/GIDとは異なる権限管理を行っているため、Docker Desktopを使用している場合、多くはDocker側が自動的にアクセス権限を調整してくれます。

ただし、**WSL2（Windows Subsystem for Linux）** 上でソースコードを管理し、そこからDockerコンテナを起動している場合はLinuxと同じ扱いになります。その場合は、WSL2のターミナル（Ubuntuなど）を開き、上記の `id` コマンドを実行してUIDとGIDを確認してください。
